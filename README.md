## 制作の背景

私は一つのことに集中して取り組むことが得意な一方で、複数のタスクを同時に整理しながら進めることに課題を感じていました。

そこで、自分自身のタスク管理力を高めることを目的に、日々の予定や作業を分かりやすく管理できるアプリを制作しました。

---

## 概要

ShibaToDo は、React で制作したタスク管理アプリです。

ログイン後に自分のタスクを確認し、タスクの追加・編集・削除・詳細表示・完了切り替えまで行うことができます。

単にCRUD機能を実装するだけでなく、ログイン状態の保持、再読み込み時のデータ復元、タスク完了トグルの操作感など、実際に使う場面を想定して改善を重ねました。

## 要件定義

### 機能要件

- アカウント作成、ログイン、ログイン状態の保持、ログアウトができる
- 再アクセス時にも認証状態を復元できる
- ユーザーごとに自分のタスクのみを管理できる
- タスクの追加、編集、削除、詳細表示、完了切り替えができる
- ホーム画面で未完了タスクやタスク数のサマリーを確認できる
- ホーム画面で期限が近いタスクを確認できる
- 未ログイン時とログイン時でヘッダー表示を切り替える
- 認証状態が解除された場合、トップ画面へ戻る
- ホーム上のタスクカードをダブルクリックすると詳細画面へ遷移できる

### 非機能要件

- Figmaで設計した画面をもとにUIを実装する
- 未ログイン時はアプリ概要とログイン導線を表示する
- 再読み込み時にも必要なデータを再取得し、画面を復元できる
- 直感的に操作できるUIにする
- 専用のエラー画面・ローディング画面を表示する
- UIコンポーネント、API関数、状態管理を分ける
- 同系統のカラーを使い、画面デザインに統一感を持たせる
- 優先度や状態を色付きラベルで見やすくする

## 工夫した点

### タスク完了トグルの操作感を改善

タスクの完了状態を切り替える際、先にUIを更新し、その後APIと同期する形にしました。

これにより、ユーザーが操作した直後に画面へ反映されるため、よりスムーズな操作感を実現しています。

また、API更新に失敗した場合は表示を元に戻すことで、UIとデータの不一致を防ぐようにしました。

### AIを補助として活用

AIは、Tailwind CSSのクラス設計、静的UIのたたき台作成、不明点の確認、要件整理の壁打ちなどに活用しました。

一方で、状態管理、画面遷移、API連携、バグ修正、挙動改善、実装の調整は、自分自身で理解しながら進めました。

AIに任せきりにするのではなく、必要な部分だけ補助として活用し、実装内容を理解した上で改善を重ねた点を意識しました。

### フロントエンドとAPI処理を分離

フロントエンドのUI処理と、json-serverとの通信処理を分けて実装しました。

API通信処理を専用の関数として分離することで、コンポーネント内の処理を整理し、UI・状態管理・データ取得の役割を分かりやすくしました。

### 機能だけでなく使い心地まで意識

単にCRUD機能を実装するだけでなく、実際に使ったときの操作感や、意図しない動作が起きにくいことも意識しました。

表示速度、画面遷移、再読み込み耐性、タスク完了トグルの反応など、ユーザーがストレスなく使えることを重視して改善しました。

## 構成のポイント

このアプリでは、コンポーネント、状態管理、API通信処理を分けて、管理しやすい構成を意識しました。

| 分類                                   | 役割                                                               |
| -------------------------------------- | ------------------------------------------------------------------ |
| App / AppLayout                        | アプリ全体のルーティングと共通レイアウトを管理                     |
| Home                                   | ログイン状態に応じて表示内容を切り替えるトップ画面                 |
| Tasks / TaskItem / TaskShow / TaskForm | タスク一覧、単体表示、詳細表示、作成・編集フォームを役割ごとに分割 |
| apiTasks / apiUsers                    | json-serverとの通信処理をUIコンポーネントから分離                  |
| store / tasksSlice / accountsSlice     | タスク状態と認証状態を分けて管理                                   |

### 構成で意識したこと

Reactコンポーネントには、なるべく画面表示に関する処理を中心に持たせ、API通信や状態管理の処理は別ファイルに分けることを意識しました。

これにより、どのファイルが何を担当しているのかを把握しやすくし、後から修正や機能追加をしやすい構成を目指しました。

特に、タスク関連の処理は一覧表示、詳細表示、フォーム、API通信、Reduxの状態管理に分けることで、責務を整理しました。

## まとめ

このアプリでは、CRUD機能の実装に加え、ログイン状態の保持、 再読み込み時のデータ復元、タスク完了トグルの操作感改善など、 ユーザーが実際に使う場面を想定して作り込みました。

特に苦労した点は、再読み込みによってReduxのステートが初期化され、 画面表示やログイン状態、タスク詳細の表示などに不具合が発生したことです。 そのため、再読み込み後も必要な情報を復元できるように改善しました。

実際にWebアプリを制作する中で、これまで学んできたReact、Redux、 React Router、非同期処理などの知識がまだ曖昧だったことを実感しました。 一方で、エラーの原因を一つずつ調べながら修正を重ねることで、 それぞれの知識がつながり、アプリ全体のデータの流れをより明確に理解できました。

今後の課題としては、RouterのactionがReduxストアを直接操作している設計を見直し、 より保守しやすい構成に改善していきたいと考えています。
